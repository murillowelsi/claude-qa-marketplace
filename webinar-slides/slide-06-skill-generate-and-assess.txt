  /generate-test-cases
  --------------------
  THE PROBLEM: Writing test cases manually from AC is repetitive,
  time-consuming, and often misses edge cases.

  WHAT IT DOES:
  For each acceptance criterion, the agent derives four test categories:
    - Positive (happy path) — expected outcome when everything is correct
    - Negative (error/rejection) — invalid input, wrong state
    - Edge case — boundary values, empty states, max limits, concurrency
    - Non-functional — performance, security, accessibility

  Each test case includes:
    - A clear, human-readable name
    - Pre-conditions
    - Numbered steps
    - Expected result (pass/fail observable outcome)

  Output is saved to: qa-output/test-cases/<ticket>-test-cases-<date>.md

  COMMAND: /generate-test-cases PROJ-123

  -  -  -  -  -  -  -  -  -  -  -  -  -  -  -  -  -  -  -  -  -  -  -  -

  /assess-risk
  ------------
  THE PROBLEM: There's never enough time to test everything.
  Teams test in random or habitual order, missing critical paths.

  WHAT IT DOES:
  Applies ISTQB risk-based testing: Risk = Likelihood x Impact
  Scores each area from 1-5 on both dimensions.

  Output: A ranked risk matrix + recommended execution order in 4 phases:
    Phase 1 — Critical (test before any release)
    Phase 2 — High (required for release sign-off)
    Phase 3 — Medium (if time allows)
    Phase 4 — Low (regression / exploratory)

  Output saved to: qa-output/risk-assessments/<topic>-risk-<date>.md

  COMMAND: /assess-risk (after /generate-test-cases, or standalone)

  -  -  -  -  -  -  -  -  -  -  -  -  -  -  -  -  -  -  -  -  -  -  -  -

  COMBINED IMPACT:
  50%+ reduction in manual test case writing time.
  Risk-based prioritization replaces guesswork with documented rationale.
